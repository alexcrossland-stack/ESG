#49 - First-Time Activation Audit & Fix

## Objective

Perform a thorough end-to-end QA and product validation pass of the platform from the perspective of a new small business user with no prior ESG knowledge.

This is a beta-stage platform. The task is not limited to finding broken code. It must also surface:
- hidden assumptions
- missing guidance
- weak defaults
- unexplained ESG jargon
- places where the UI expects the user to already understand implementation logic

Actively try to break the platform, not just confirm happy paths.

This task validates and hardens what already exists. It does not redesign the onboarding architecture or add feature sets covered elsewhere.

## Required operating approach

Work in this order.

### Phase 1 — Understand the system
Before changing any code:
- Review the relevant frontend pages, routes, backend endpoints, auth/session handling, permission guards, and supporting components/services
- Understand what the UI implies should happen versus what the backend actually enforces
- Note all mismatches before starting fixes

### Phase 2 — Full audit pass
Run the six high-risk journeys in the order listed below.
Do not stop at the first issue.
Compile a full bug/friction list first.

### Phase 3 — Fix pass
Fix all Critical and High severity issues discovered in the audit.
Document Medium and Low issues without fixing them unless they are trivial and clearly safe.

### Phase 4 — Regression pass
Re-run all affected journeys after fixes.
Confirm no regressions.
Add or extend Playwright coverage for critical paths.

## High-risk journeys — test in this order

### 1) Signup / login
Test:
- registration form
- duplicate email
- weak password
- invalid email format
- login with wrong credentials
- login with correct credentials
- session persistence after refresh
- logout
- session expiry / invalid session behavior
- MFA setup or MFA challenge if triggered

Check for:
- broken validation
- generic or misleading error messages
- auth loops
- silent failures
- raw API errors surfaced to the user
- inability to recover from MFA-related issues

### 2) Onboarding / company setup
Test:
- all 7 wizard steps
- minimum viable input at each step
- skipping optional steps
- Quick Start path
- back navigation
- page refresh mid-flow
- browser back/forward behavior
- final completion state
- action plan generation if included in the flow

Audit all visible copy as a non-ESG user:
- headings
- labels
- helper text
- button labels
- tooltip text

Check for:
- unclear purpose of each step
- ESG jargon without explanation
- missing defaults
- broken step transitions
- ambiguous completion states
- post-onboarding dead ends
- Quick Start creating a broken or partial state

### 3) First ESG data entry
Test:
- finding the Metrics Library
- understanding metric categories as a new user
- adding a first data point
- saving/editing the data point
- attaching evidence
- revisiting the saved item

Check for:
- unfamiliar terms not explained
- unclear units or expected inputs
- missing defaults
- save failures
- evidence upload/linking issues
- confusing category structure
- weak empty states
- inability to tell what to do next

### 4) Dashboard comprehension
Test the first dashboard experience after onboarding.

Check:
- ESG score ring
- activation checklist
- data quality card
- all summary cards/panels
- every CTA button/link on the dashboard

Answer these questions:
- Does the dashboard make sense with little or no data?
- Are score and status values explained?
- Is any jargon unexplained?
- Do checklist items reflect real progress?
- Do all CTA buttons route correctly?
- Are there blank-looking panels or misleading zero states?

Fix:
- broken CTAs
- broken checklist logic
- confusing empty states
- ambiguous score presentation for first-time users

### 5) Report / export generation
Test:
- generating a report for the first time
- understanding framework selection options
- preview flow if available
- export/download flow if available

Check for:
- unexplained framework choices
- failed generation
- broken previews
- stale or missing data in output
- download failures
- poor first-run guidance

### 6) Role and permission enforcement
Create or use a viewer-role user and test against an admin user.

Verify:
- viewer cannot perform write actions
- admin-only routes reject non-admins
- buttons hidden in the UI are also enforced in the API
- tenant isolation is enforced
- user from company A cannot see or modify company B data

Check for:
- UI-only permission checks
- missing API enforcement
- route guard inconsistencies
- insecure direct object access
- tenant data leakage

## Cross-cutting requirements

### Form validation and error handling
Across all tested flows:
- validation errors must be specific
- validation errors must appear next to the relevant field where applicable
- known failure modes must not rely on generic “something went wrong” toasts
- API error responses must surface meaningful user-friendly messages
- remove silent failures and blank states caused by handled errors

### Copy and comprehension
This platform is for SME users with no prior ESG knowledge. During the audit, flag and improve:
- jargon-heavy wording
- headings that do not explain why the step matters
- labels that assume ESG knowledge
- empty states with no next-step CTA
- confusing defaults

### Beta watchpoints
Pay special attention to:
- “maturity”
- “framework”
- “evidence”
- “metric category”
- “scope 1/2/3”
- “GRI”
- “TCFD”
- any place the UI assumes the user already knows the next action
- any minimum-input path that produces a broken or bewildering state
- any mismatch between UI permissions and API permissions

## Severity definitions

- Critical — Blocks signup, onboarding, first use, report generation, core data entry, auth, security, or tenant isolation
- High — Major friction or broken behavior in important flows, but not total blockage
- Medium — Noticeable bug, confusing UX, copy issue, or weak empty state that does not fully block progress
- Low — Minor polish, cosmetic inconsistency, or low-impact friction

## Done looks like

### Technical
- all six high-risk journeys pass end-to-end without errors, broken navigation, or silent failures
- form validation errors are specific, actionable, and field-adjacent where appropriate
- API error responses surface meaningful messages to the user
- role and permission rules are enforced in both UI and backend
- Quick Start seeds data correctly and lands the user in a functional dashboard state
- MFA flow does not block access without a clear recovery path

### UX / Product
- every onboarding step has a plain-English heading and sub-copy explaining why it matters
- ESG-specific terms used in the UI have inline helper text or tooltip support where already available in the design system
- every key page first visited after onboarding has a non-broken empty state and a clear CTA:
  - Metrics
  - Policies
  - Evidence
  - Reports
  - Carbon Calculator
- dashboard activation checklist reflects real progress
- all dashboard CTAs navigate correctly
- no onboarding path ends in a blank or ambiguous success state

## Deliverables

### 1. Structured bug report
Use these columns:
- ID
- severity
- type (bug / UX / copy / permission / security)
- area
- reproduction steps
- expected behavior
- actual behavior
- root cause
- fix implemented
- test coverage added
- status

### 2. Commands and test scripts run
Include:
- all commands run
- all Playwright scripts used
- any manual test setup data created

### 3. Code change summary
Include:
- what files changed
- what was fixed
- why each change was necessary

### 4. Remaining risks / known issues
List all unresolved Medium/Low issues and anything deferred.

### 5. Hardening recommendations
Give practical recommendations for improving activation, usability, error resilience, and permission safety.

### 6. Final verdict
Answer explicitly:
- Can a new small business user with no ESG knowledge successfully set up and use the platform without help?
- If not, what still blocks that outcome?

## Out of scope
- Building new onboarding steps or redesigning the wizard structure
- Adding new contextual guidance panels beyond small copy/helper/tooltip fixes
- Stripe billing integration
- Email delivery infrastructure
- Admin-side monitoring and ops configuration
