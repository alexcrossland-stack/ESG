Identity Hardening, SSO Readiness & GDPR Data Rights

What & Why

Extend the security foundation into identity maturity and regulatory readiness. This phase introduces multi-factor authentication (MFA), prepares the platform for SSO/SAML without overbuilding, and implements GDPR-aligned data rights tooling (export, deletion, consent traceability).

The goal is not full enterprise identity yet, but to:
	•	materially reduce account takeover risk (via MFA)
	•	make enterprise onboarding feasible (via SSO-ready architecture)
	•	ensure legal defensibility and customer trust (via GDPR rights tooling)

This phase builds directly on #12 and assumes auth, RBAC, tenant isolation, audit logging, and API key controls are already in place.

⸻

Done looks like

MFA (TOTP-based) is available for all users and enforceable for admins and optionally company-wide. Users can enable MFA via authenticator apps (e.g. Google Authenticator, Microsoft Authenticator). Backup/recovery codes are generated and stored securely (hashed). MFA is required at login once enabled, and recovery flow is available if the device is lost.

MFA enforcement policies exist:
	•	optional (user-level opt-in)
	•	required for admins
	•	required for entire company (configurable by admin)

SSO readiness is implemented at the architecture level:
	•	user identity model supports external identity providers
	•	auth system supports linking external identity to internal user
	•	login flow is abstracted to allow future SAML/OAuth providers without refactor
	•	no hard dependency on local password login

A minimal SSO configuration model exists (no full SAML implementation required yet), including:
	•	identity_provider table or equivalent
	•	support for storing provider type, issuer, client_id, and metadata placeholders
	•	ability to map domain → SSO requirement (future-ready, not enforced yet unless simple)

GDPR data rights tooling is implemented:

Data export:
	•	company admin can export all company data (structured JSON/CSV bundle)
	•	user can export their personal data
	•	export includes metrics, reports, evidence metadata, and user/account data (excluding secrets)
	•	export jobs are logged and auditable

Data deletion:
	•	user can request account deletion
	•	admin can delete company (soft-delete or staged delete)
	•	deletion process removes or anonymises personal data while preserving audit integrity where required
	•	deletion actions are logged

Data ownership clarity:
	•	all records clearly tied to company_id and/or user_id
	•	no orphaned personal data

Consent and legal traceability:
	•	record user acceptance of terms/privacy policy with timestamp
	•	versioned policy reference stored

Audit logs capture:
	•	MFA enable/disable
	•	MFA enforcement changes
	•	recovery code usage
	•	data export requests
	•	data deletion requests and completion
	•	SSO configuration changes (if applicable)

Admin UI includes:
	•	MFA enforcement controls
	•	user MFA status visibility
	•	data export trigger
	•	company deletion controls (with safeguards)

Out of scope

Tasks

MFA (TOTP) implementation

Implement TOTP-based MFA using a standard library (e.g. speakeasy or otplib).
	•	Add user fields:
	•	mfa_enabled (boolean)
	•	mfa_secret (encrypted)
	•	mfa_backup_codes_hash (hashed array)
	•	mfa_enabled_at
	•	MFA setup flow:
	•	generate secret
	•	display QR code + manual code
	•	require user to verify TOTP before enabling
	•	generate backup codes (shown once)
	•	MFA login flow:
	•	after password validation, require TOTP if enabled
	•	validate token with time window tolerance
	•	allow backup code as fallback (one-time use, then invalidate)
	•	MFA management:
	•	enable / disable MFA (requires password + TOTP confirmation)
	•	regenerate backup codes
	•	MFA enforcement:
	•	add company setting: mfa_policy = optional | admin_required | all_required
	•	enforce at login (block access until MFA set up if required)
	•	Audit events:
	•	MFA enabled/disabled
	•	backup code used
	•	failed MFA attempts

Acceptance:
	•	User can enable MFA and log in with TOTP
	•	Backup codes work once and are invalidated
	•	Enforced MFA blocks login until configured

SSO readiness (architecture only)

Prepare identity model and auth flow for future SSO integration.
	•	Add identity provider model:
	•	id
	•	company_id
	•	provider_type (saml | oidc)
	•	issuer / domain
	•	client_id
	•	metadata_json (for future config)
	•	is_active
	•	Extend user model:
	•	external_id nullable
	•	identity_provider_id nullable
	•	Refactor auth service:
	•	separate “identity verification” from “session creation”
	•	allow multiple auth methods:
	•	password
	•	external provider (future)
	•	Add placeholder endpoints:
	•	initiate SSO login (stub)
	•	handle SSO callback (stub)
	•	Domain mapping (optional but preferred):
	•	allow company to define email domain
	•	future-ready enforcement for SSO-required domains
	•	Audit events:
	•	identity provider created/updated
	•	SSO config changes

Acceptance:
	•	Codebase supports multiple auth methods without refactor
	•	Identity provider records can be created and stored
	•	No regression to existing login flow

GDPR Data Export

Implement structured data export capability.
	•	Export scope:
	•	company data (admin-triggered)
	•	user personal data (self-triggered)
	•	Include:
	•	user profile data
	•	company profile
	•	metrics and metric values
	•	reports
	•	evidence metadata (not necessarily raw files unless feasible)
	•	audit-relevant records tied to user/company
	•	Exclude:
	•	passwords
	•	API key raw values
	•	secrets
	•	Format:
	•	JSON bundle (primary)
	•	optional CSV for tabular data
	•	Delivery:
	•	generate export asynchronously
	•	store temporarily
	•	provide secure download link (expiring)
	•	Audit:
	•	log export request and completion

Acceptance:
	•	Admin can export full company dataset
	•	User can export personal data
	•	Export is complete and structured

GDPR Data Deletion

Implement controlled deletion flows.
	•	User deletion:
	•	user can request account deletion
	•	anonymise or delete personal data
	•	preserve audit logs with anonymised actor reference if required
	•	Company deletion:
	•	admin-triggered
	•	staged process:
	•	mark as pending deletion
	•	delay window (optional safeguard)
	•	perform cascade delete/anonymisation
	•	Data handling rules:
	•	remove PII where required
	•	retain non-personal aggregated data if needed
	•	ensure no dangling references
	•	Audit:
	•	log deletion request
	•	log completion

Acceptance:
	•	User deletion removes or anonymises personal data
	•	Company deletion removes all tenant data safely
	•	Audit trail remains intact

Replit build instruction

Extend the platform’s identity and compliance layer by implementing TOTP-based MFA with enforcement policies, preparing the authentication architecture for future SSO integration without breaking current login flows, and adding GDPR-compliant data export and deletion tooling.

Reuse existing auth, user, and audit systems. Add new models for identity providers, export jobs, and deletion requests. Ensure MFA, export, and deletion flows are secure, auditable, and do not expose sensitive data.

All new features must integrate with existing RBAC, tenant isolation, and audit logging. MFA and GDPR actions must be enforced server-side and logged. Avoid overengineering SSO — focus on making the system extensible rather than fully implemented.

Deliver in clear sections: MFA, SSO readiness, data export, data deletion, consent tracking, and UI updates.
