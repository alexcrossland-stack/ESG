#45 - Session Security & Step-Up Authentication

What & Why

Build on the security and MFA foundation by hardening session handling and requiring re-authentication for high-risk actions. At this stage, the most likely real-world account compromise path is not sophisticated identity federation failure; it is stolen sessions, unattended devices, weak session lifecycle controls, or a privileged user performing a destructive action without recent authentication.

This phase improves account and admin protection without adding major operational overhead. It is appropriate for an SME-focused SaaS platform expecting up to 100 customers in the first 6 months and is designed to reduce practical risk while remaining simple to operate.

Done looks like

Users can view a list of active sessions in their account settings, including device/browser summary, approximate IP/location if available, last activity, and current session marker. Users can revoke individual sessions or log out all other sessions. Session lifecycle controls exist with explicit idle timeout and absolute session expiry. High-risk actions require step-up authentication using password re-entry and MFA where enabled. Step-up authentication is enforced before API key creation, company deletion, account deletion, admin role changes, MFA disable, backup code regeneration, and GDPR export/deletion actions. Super-admin and admin accounts have stricter step-up coverage than ordinary users where appropriate. Session revocation invalidates the targeted session server-side immediately. Audit logs capture session creation, session revocation, logout-all-other-sessions, step-up prompts, step-up failures, and successful high-risk action confirmations. Session-related controls do not break normal day-to-day navigation or create excessive user friction.

Out of scope

Passkeys or WebAuthn
Device trust / remember-this-device MFA bypass
Advanced geo-risk or impossible-travel detection
Full user-facing device fingerprinting
Continuous adaptive authentication
Consumer-style account recovery workflows beyond the existing MFA backup code model

Tasks

Session model and storage hardening — Confirm the existing auth/session architecture supports explicit server-side invalidation of individual sessions. Add or extend a session store so sessions have stable identifiers, created_at, last_seen_at, expires_at, revoked_at, user_id, and lightweight user-agent / IP metadata where feasible. Ensure the current session can be distinguished from other sessions.

Session lifecycle controls — Add an idle timeout and an absolute session lifetime. Idle timeout should be reset on valid activity and enforced on protected routes. Absolute lifetime should force re-authentication even if the session remains active. Make both values configurable via environment or central config and document defaults suitable for this stage.

User session management UI — Add an account security/session management section in user settings showing active sessions. Display current session, last activity, created time, and rough device/browser metadata. Allow revoke single session and revoke all other sessions. Revoke actions must invalidate server-side state immediately.

Step-up authentication framework — Introduce a reusable step-up auth mechanism that can be applied to high-risk routes. A recent-auth token or equivalent server-side marker may be used after successful password re-entry and MFA verification where enabled. Step-up status must expire after a short window and be scoped to the authenticated user/session.

High-risk action enforcement — Require step-up authentication before API key creation, company deletion, account deletion, MFA disable, backup code regeneration, admin role elevation or change, and GDPR export/deletion actions. Ensure backend enforcement is mandatory and frontend UI is not the only control. For users with MFA enabled, step-up should require both password and TOTP or backup-code fallback as appropriate.

Audit logging — Emit audit events for session created, session revoked, logout all other sessions, step-up success, step-up failure, and high-risk action confirmed after step-up. Include actor, session id if available, IP, user-agent, target action, and timestamp.

Admin protection rules — Apply stricter defaults or broader step-up coverage for admin and super-admin actions, especially role changes, company deletion, and privileged settings changes. Ensure these controls are enforced server-side.

Acceptance criteria

Users can see active sessions and revoke them individually. Users can log out all other sessions without terminating their current session unless explicitly requested. Idle timeout and absolute session expiry are enforced. High-risk actions fail without recent step-up authentication. Step-up authentication succeeds with password and MFA where enabled. Revoked sessions cannot continue using protected routes. All session and step-up security events are captured in the audit log.

Replit build instruction

Implement session lifecycle hardening and step-up authentication across the platform. Reuse the existing auth, MFA, RBAC, and audit foundations. Add server-side session identifiers, expiry and revocation support, a user-facing active sessions view, and a reusable step-up authentication mechanism for high-risk actions. Enforce all controls in the backend, keep the UX lean, and prioritise immediate server-side invalidation and clear auditability over advanced device intelligence.

#46 - Monitoring, Alerts & Incident Basics

What & Why

Convert audit logging from passive record-keeping into practical operational monitoring. At this stage, the platform does not need a full SIEM or enterprise detection stack. It does need simple, reliable, actionable alerts for suspicious activity and a lightweight incident response foundation so issues are detected quickly and handled consistently.

This phase is appropriate for an early-stage SaaS platform because it focuses on threshold-based alerting, operational runbooks, and visibility for the highest-risk events without adding heavy infrastructure.

Done looks like

Threshold-based alerts exist for repeated failed logins, repeated failed MFA attempts, repeated failed agent/API key authentication, unusual export/delete activity, admin role changes, company deletion requests, and super-admin actions. Alerts are delivered to a practical operational channel such as email, Slack webhook, or equivalent. Alerting is rate-limited and deduplicated enough to avoid spam storms. Internal dashboards or admin views show recent security events and alert history at a lightweight level. A documented incident response runbook exists covering account compromise, leaked API key, suspicious export activity, accidental deletion, cross-tenant access concern, malicious file upload concern, and secret exposure. The platform can disable or revoke compromised sessions, API keys, or affected user access quickly. Alert logic does not block the underlying application path if the notification channel is unavailable; alert failures are logged internally. Monitoring remains proportionate and low-maintenance.

Out of scope

Full SIEM integration
24/7 SOC workflow
Automated threat hunting
Machine-learning anomaly detection
Pager escalation tooling
Formal forensic pipeline
External MDR services

Tasks

Security event classification — Define a minimal set of security-relevant event categories based on the existing audit log: auth failures, MFA failures, privileged role changes, API key failures, export/delete activity, session revocations, company deletion requests, and super-admin actions. Ensure event types are consistent and queryable.

Alert rule engine — Add a lightweight alert evaluation layer that can trigger notifications from recent audit events and thresholds. Suggested initial rules: repeated failed logins by account or IP in a short window, repeated failed MFA attempts by account, repeated failed API key auth, multiple export requests in a short time window, admin role change, company deletion request, super-admin privilege change or critical action, and unusual volume of denied access attempts. Keep rules configurable and easy to extend.

Notification delivery — Implement one or two practical delivery channels only, such as email and Slack webhook. Notifications must be best-effort: alert delivery failure must not break the primary app action. Log failed alert delivery internally. Add simple deduplication and cooldown logic to prevent alert floods.

Security overview UI — Add a lean admin or super-admin security overview panel showing recent high-severity alerts, recent failed logins, recent failed MFA attempts, recent export/delete requests, and recent admin/super-admin actions. Keep it read-only and operational rather than analytical.

Incident runbooks — Commit lightweight internal docs for the most important scenarios: suspected user account compromise, suspected admin compromise, leaked API key, suspicious export/delete request, secret exposure, cross-tenant access concern, accidental deletion, and malicious upload concern. Each runbook should specify how to detect, what to disable or revoke first, what logs to inspect, and what manual communication or remediation steps to take.

Containment helpers — Ensure the platform already supports the practical containment actions needed by the runbooks: revoke sessions, revoke API keys, disable accounts if necessary, and pause or block dangerous follow-on actions where reasonable. Add missing admin controls only where lightweight and necessary.

Audit and alert logging — Log when an alert rule fires, when a notification is sent, when delivery fails, and when an alert is acknowledged if you add acknowledgement support. Ensure alert history is queryable.

Acceptance criteria

Repeated failed login activity triggers an operational alert. Repeated MFA failures trigger an operational alert. Admin role changes and company deletion requests trigger alerts. Export/delete spikes trigger alerts. Alert delivery failure does not break the app action. A super-admin security overview shows recent alert-worthy events. Incident runbooks are committed and usable.

Replit build instruction

Implement lightweight security monitoring and alerting using the existing audit/event foundation. Add threshold-based alert rules, practical delivery channels, a simple operational visibility panel, and internal incident-response runbooks. Keep the design intentionally lightweight and low-maintenance: prioritize actionable signals, best-effort delivery, and fast containment over complex detection tooling.

#47 - Backup Verification, File Safety & Data Retention Basics

What & Why

Improve resilience against the most practical early-stage risks: accidental data loss, unsafe uploads, and unnecessary long-lived sensitive data. At this stage, backup existence alone is not enough; restore capability must be verified. File validation alone is not enough; uploaded files should move toward a safer lifecycle. GDPR export and operational file generation also introduce short-lived sensitive artifacts that should not remain indefinitely.

This phase is appropriate for a platform expecting up to 100 customers because it materially improves resilience and reduces common operational/security risk without requiring enterprise-grade storage architecture.

Done looks like

A documented and repeatable restore verification process exists for the database and critical file storage. Restore tests are performed on a scheduled basis and logged. Recovery expectations are documented with simple target RTO and RPO values suitable for the business. Uploaded evidence files have a clearer safety lifecycle, including mandatory MIME/extension checks, optional scan/quarantine workflow if implemented, and explicit status handling where needed. High-risk file types remain blocked. Export artifacts are automatically deleted after expiry. Short-lived generated files and temporary artifacts have retention rules. Basic data retention rules exist for key categories such as export bundles, deletion staging records, and temporary uploads. The platform reduces unnecessary long-lived storage of sensitive operational artifacts while keeping user-facing behaviour simple and auditable.

Out of scope

Full DLP tooling
Advanced sandboxing of uploaded files
Content disarm and reconstruction
Long-term archive platform
Formal records-management programme
Cross-region backup architecture
Customer-defined retention policies

Tasks

Backup restore verification — Add a documented restore verification process for the primary database and critical file/evidence storage. Define a simple schedule for restore testing, such as monthly or quarterly. Record each test outcome, timestamp, operator, and notes in a lightweight log or internal doc. Document target RTO and RPO values suitable for the stage of the business.

Restore tooling/documentation — Improve the existing backup/restore runbook so it includes a practical restore rehearsal path, not just theoretical steps. Document prerequisites, test environment expectations, validation checks after restore, and rollback/cleanup steps for rehearsal environments.

Export artifact retention — Ensure GDPR export files and other generated sensitive bundles are auto-deleted after expiry. Add cleanup jobs if needed. Ensure expired download links do not continue to grant access. Log cleanup actions and failures.

Temporary file and artifact retention — Identify short-lived file categories created by the app, such as exports, temporary upload staging files, or generated report bundles. Add default retention rules and scheduled cleanup. Ensure cleanup is idempotent and safe to retry.

File safety lifecycle — Build on the earlier file upload controls by introducing a clearer file status model where appropriate, such as pending, available, quarantined, rejected, deleted, or similar. If malware scanning is straightforward, implement a simple scan/quarantine workflow. If not, add the status model and hooks so scanning can be added later without redesign. Continue blocking executable and script-like files outright.

File access logging — Ensure sensitive file access and export download activity is captured in the audit log with actor, file/export identifier, timestamp, and outcome where practical.

Retention basics by data type — Define and document minimal retention rules for sensitive operational data such as export artifacts, deletion request staging data, backup verification logs, and temporary report bundles. Keep this operationally light and focused on reducing unnecessary risk rather than building a full governance framework.

Acceptance criteria

A database restore rehearsal process is documented and performed at least once in a testable manner. Critical file storage restore steps are documented and verified where feasible. RTO and RPO targets are documented. Export files expire and are automatically deleted after expiry. Temporary generated files are cleaned up on schedule. File status handling exists and supports future quarantine/scanning workflows. Sensitive file and export access are auditable.

shared/schema.ts

docs/data-retention.md

Replit build instruction

Implement a lightweight resilience and file-safety phase focused on restore verification, sensitive artifact cleanup, and clearer file lifecycle handling. Build on the existing backup, file upload, audit, and GDPR export foundations. Prioritize practical restore testing, automatic cleanup of expiring artifacts, clear retention rules for short-lived sensitive files, and a simple file status model that keeps the current system safe now and extensible later.
