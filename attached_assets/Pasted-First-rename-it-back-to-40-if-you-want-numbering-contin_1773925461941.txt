First, rename it back to #40 if you want numbering continuity with the rest of your roadmap. Replit changed it to #12, which is fine structurally, but it will break sequencing if your wider plan is already using the later phase numbers.

Second, slightly tighten the tenant-isolation wording so Replit does not interpret it as only a documentation exercise. The phrase about committing docs/security-isolation-audit.md is good, but the implementation requirement should clearly say that insecure routes must be refactored, not just reviewed.

Third, in the file upload section, I would avoid making “first 512 bytes / magic bytes” feel mandatory unless you know they are comfortable implementing it cleanly in the current stack. Keep MIME + extension validation mandatory, and treat content sniffing as preferred rather than a blocker if needed.

Fourth, in the rate limiting section, “per-user and per-company limits on report generation, CSV import, and AI/agent endpoints” is a good idea, but it may broaden the phase a bit. If you want this phase to land faster, keep auth/reset/upload mandatory and make the broader endpoint throttling part of the same phase only where low effort.

Fifth, add one explicit requirement that audit logging must never block the user-facing action. In other words, if audit logging fails, the core business action should usually still complete and the audit failure should be logged internally. That prevents brittle writes.

Below is the version I would use as the final copy-and-paste spec.

#40 - Security Foundation: Auth, Tenant Isolation & Audit

What & Why

Harden the platform with a practical SaaS security baseline that addresses the highest-likelihood and highest-impact risks before real users and data enter the system: account compromise, cross-tenant data leakage, insecure file access, weak agent/API key controls, and poor auditability. This phase replaces the earlier narrower security hardening scope with a full enforced baseline. The governing principle is simple: never trust the frontend. Access, roles, company ownership, site ownership, API key scope, and file access must all be enforced server-side on every sensitive route. This phase should create a credible minimum security baseline for real customer use, future procurement scrutiny, GDPR readiness, and later SOC 2-aligned work.

Done looks like

All protected routes use a central requireAuth middleware and reject unauthenticated access by default; only explicitly marked public routes are open. Session and auth cookies in production use httpOnly: true, secure: true, and sameSite: lax or stricter. Session expiry is explicit. Logout invalidates the session server-side. Password and login flows do not leak whether an email exists, using consistent error messages and safe comparison patterns. Reusable requireRole(…roles) and requireSuperAdmin() guards are applied to all company settings, team management, billing, metric approval, and super-admin routes. Frontend hiding is never the only protection; forbidden access returns 403. Every tenant-owned read and write route is scoped by company_id sourced from session context, never from request body or params. Any siteId in a request is validated as belonging to the authenticated company. Any insecure route discovered during the audit is refactored so ownership enforcement is explicit and consistent. A written isolation audit document at docs/security-isolation-audit.md is committed listing audited routes, findings, and fixes made. Agent API keys are generated with cryptographically secure randomness. Only key_hash and key_prefix are stored; the raw key is shown once at creation. Revoked or expired keys fail auth immediately. Key list endpoints never return raw key material. File uploads validate both extension and MIME type, block executable or script-like types, enforce file size limits, and are stored privately rather than in public static paths. Retrieval requires authenticated authorised access through a controlled backend endpoint. Download and delete routes enforce tenant ownership. An audit log captures login success and failure, password reset, role changes, team invite and remove, company settings changes, subscription changes, API key create and revoke, agent escalation and actions, metric approval and rejection, evidence upload and delete, and super-admin actions. Each event records actor type, actor ID, action, target type and ID, IP, user-agent, metadata, and timestamp where available. Audit logging must be best-effort and must not unnecessarily block the primary business action if logging fails. Login, password reset, invite resend, and upload endpoints are rate-limited. 429 is returned on threshold breach. Internal workers are not inadvertently blocked. Security headers are set, including Strict-Transport-Security in production, X-Content-Type-Options: nosniff, Referrer-Policy, X-Frame-Options or CSP frame-ancestors, and a baseline Content-Security-Policy. The global error handler logs full detail server-side and returns sanitised messages in production. Health endpoints expose only safe status and do not leak secrets or internals. .env.example documents all required variables. No hardcoded secrets remain in the repo. Secret rotation requires no code changes. A deployment checklist and backup/restore documentation are committed. Company admin UI shows recent audit events, active and revoked API key history, and API key management with create, scope selection, one-time raw key display, revoke, and last-used timestamp. Super-admin UI shows platform-wide audit visibility, recent failed auth attempts, recent privileged actions, and recent subscription and admin changes.

Out of scope

SSO or SAML integration. Full MFA rollout. Full SOC 2 policy pack. WAF or CDN automation. SIEM integration. Advanced anomaly detection. Customer-managed encryption keys. Full data residency controls. Penetration testing or external security audit. Mandatory ClamAV installation.

Tasks

Auth and session hardening — Create a central requireAuth middleware used across all protected routes. Harden session and cookie config for production with httpOnly, secure, and sameSite. Make session expiry explicit. Ensure logout invalidates session state server-side. Normalise login and password-reset error messages to prevent email enumeration. Confirm passwords are hashed with bcrypt or argon2 with adequate cost.

Backend RBAC guards — Add requireRole(…roles) and requireSuperAdmin() middleware in server/permissions.ts. Apply them to company settings, team and role management, billing, metric approval and rejection, framework configuration if admin-restricted, and super-admin platform routes. All forbidden access must return 403. Frontend guards alone must not be the only protection.

Tenant isolation and site ownership enforcement — Audit every route that reads or writes tenant-owned data to confirm company_id is always sourced from session context. Fix any routes that trust client-supplied companyId. Validate siteId ownership on every request that includes one. Scope all record updates and deletes by both id and company_id where applicable. Add shared helper functions so ownership enforcement is not duplicated inconsistently. Commit docs/security-isolation-audit.md listing every audited route, findings, and fixes applied.

Agent and API key hardening — Generate API keys using cryptographically secure randomness such as crypto.randomBytes. At creation, show the raw key once only; store only key_hash and key_prefix. Add or confirm schema fields name, scopes, created_by_user_id, created_at, last_used_at, expires_at nullable, and revoked_at nullable. Reject revoked or expired keys immediately. List endpoints must never return raw key material. Historical agent runs remain after revocation.

Secure file upload and evidence protection — Validate uploads by both extension and MIME type. Content-based verification is preferred where feasible, but MIME and extension validation is mandatory. Block executable and script-like file types. Enforce per-file max size and request size limits. Sanitise or replace original filenames. Store files privately, not in public static paths. All retrieval must go through a controlled authenticated backend endpoint that enforces tenant ownership. Apply the same ownership check to file delete routes.

Audit logging — Add or extend the audit_logs table with fields id, company_id nullable, actor_type, actor_user_id nullable, actor_agent_id nullable, action, target_type nullable, target_id nullable, ip_address nullable, user_agent nullable, metadata jsonb, and created_at. Create an auditLog() service function and instrument all security-relevant events: login success and failure, password reset, role changes, team invite and remove, company settings changes, subscription changes, API key create and revoke, agent escalation and action, metric approval and rejection, evidence upload and delete, and super-admin actions. Audit logging should be best-effort and should not unnecessarily fail the parent action if the log write itself fails.

Rate limiting and abuse protection — Add server/rate-limit.ts with strict rate limits on login, password reset, and invite resend, and moderate limits on uploads. Add broader endpoint throttling on report generation, CSV import, and AI or agent endpoints where this is low-risk and straightforward in the current architecture. Return 429 on threshold breach. Ensure internal worker calls are not inadvertently blocked.

Security headers and sanitised error handling — Add security headers middleware: Strict-Transport-Security in production only, X-Content-Type-Options: nosniff, Referrer-Policy, X-Frame-Options or CSP frame-ancestors, and a baseline Content-Security-Policy. Update the global error handler to log full detail server-side and return sanitised, non-leaking messages in production. Review /health and similar endpoints to confirm they expose only safe status.

Secrets, env config, and operational documentation — Audit the codebase for hardcoded secrets and remove them. Update .env.example to document all required variables with descriptions. Separate secrets by concern, including session, JWT if used, webhook signing, email provider, and storage. Commit a deployment checklist at docs/deployment-checklist.md and a backup/restore runbook at docs/backup-restore.md.

Security admin UI — Add a Security section to the company admin UI showing recent audit events, active API key count, revoked API key history, and API key management with create, scope selection, one-time raw key display, revoke action, and last-used timestamp. Add a security and audit panel to the super-admin console showing platform-wide audit events, recent failed auth attempts, recent privileged actions, and recent subscription and admin changes.
