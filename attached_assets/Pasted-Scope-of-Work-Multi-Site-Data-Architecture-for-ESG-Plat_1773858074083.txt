Scope of Work: Multi-Site Data Architecture for ESG Platform

Objective

Extend the ESG platform so that a single business account can create and manage multiple sites (locations) within one organisation. Each site must be able to collect, store, analyse, and report ESG data independently, while also contributing to an aggregated organisation-level view.

⸻

1) Business Goal

Introduce a second structural layer beneath organisations to support real-world company structures.

Example:
	•	Organisation: ABC Manufacturing Ltd
	•	Site: Leeds Plant (Operational)
	•	Site: Birmingham Warehouse (Operational)
	•	Site: London Head Office (Office)

Each site must function as its own ESG data container.

⸻

2) Core Product Requirements

2.1 Hierarchy

User → Organisation → Sites → (Uploads, Questionnaires, AI, Reports, Metrics)

2.2 Site Types
	•	operational
	•	office
(Must be extensible)

2.3 Dual Views
	•	Organisation-level (aggregated)
	•	Site-level (isolated)

⸻

3) Functional Scope

3.1 Site Management (CRUD)

Fields:
	•	id, organisationId, name, slug, type, status
	•	country, city, address fields
	•	createdAt, updatedAt

Actions:
	•	Create, edit, archive, list, switch context
(No hard deletes)

⸻

3.2 Site Context

All ESG-relevant data must support siteId:
	•	uploads
	•	questionnaires
	•	AI outputs
	•	reports
	•	benchmarks

⸻

3.3 Data Collection

Each site independently collects ESG data.

⸻

3.4 CSV Imports
	•	Must be tied to a site
	•	Require site selection or site context
	•	Persist siteId in all records

⸻

3.5 Questionnaires
	•	Stored per organisation + site
	•	Independent completion per site

⸻

3.6 AI Analysis
	•	Runs per site
	•	Stores siteId
	•	Outputs labelled with site

⸻

3.7 Reports
	•	Site reports (single site)
	•	Organisation reports (aggregated)

⸻

3.8 Dashboards

Organisation dashboard:
	•	Aggregated KPIs
	•	Site breakdown

Site dashboard:
	•	Site-specific metrics and activity

⸻

3.9 Navigation
	•	Site switcher (Organisation + Sites)
	•	URL structure supports site context:
	•	/dashboard
	•	/sites/:siteId/dashboard

⸻

4) Technical Scope

4.1 New Table: organisation_sites

Fields:
	•	id, organisation_id, name, slug, type, status
	•	location fields
	•	created_at, updated_at

Indexes:
	•	organisation_id
	•	organisation_id + status

⸻

4.2 Add site_id

Add to ESG-related tables:
	•	uploads
	•	questionnaires
	•	reports
	•	AI outputs
	•	benchmarks

Nullable initially.

⸻

4.3 Migration Strategy

Phase 1:
	•	Add nullable site_id

Phase 2:
	•	Optional: assign legacy data to “Primary Site”

⸻

4.4 API Endpoints

New:
	•	GET /api/sites
	•	POST /api/sites
	•	GET /api/sites/:id
	•	PATCH /api/sites/:id
	•	DELETE /api/sites/:id (archive)

Update existing APIs to support site context.

⸻

4.5 Access Control
	•	Organisation-level access only (for now)
	•	Validate site belongs to organisation

⸻

4.6 Validation
	•	Enforce site ownership
	•	Prevent cross-org access

⸻

5) Frontend Scope

5.1 Site Switcher
	•	Organisation overview + site list

⸻

5.2 Site Management Page
	•	/settings/sites
	•	Table with edit/archive actions

⸻

5.3 Site-Aware Pages

Update:
	•	dashboard
	•	uploads
	•	questionnaires
	•	reports

⸻

5.4 Empty States
	•	No sites → prompt creation
	•	No data → prompt upload/input

⸻

6) Aggregation Logic

Organisation totals:
	•	Sum across sites

Expose:
	•	Site breakdown
	•	No hidden blending

⸻

7) Activity Tracking

Track:
	•	site_created
	•	csv_uploaded_to_site
	•	report_generated_for_site
	•	ai_analysis_run_for_site

Include siteId in tracking payloads.

⸻

8) Non-Functional Requirements
	•	Backward compatible
	•	Indexed for performance
	•	Maintain auditability
	•	Designed for extensibility

⸻

9) Out of Scope

	•	Complex hierarchy
	•	Advanced ESG frameworks
	•	External client portals

⸻
