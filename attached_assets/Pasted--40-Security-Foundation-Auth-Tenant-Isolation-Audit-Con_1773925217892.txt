#40 - Security Foundation: Auth, Tenant Isolation & Audit Controls

What & Why

Harden the platform with a practical SaaS security baseline that reduces the highest-likelihood and highest-impact risks: account compromise, cross-tenant data leakage, insecure file access, weak agent/API key controls, and poor auditability. This phase is not about enterprise-grade overengineering. It is about turning security from an implied intention into explicit platform behaviour enforced by middleware, route guards, storage rules, audit logs, and production-safe configuration. For this platform, the most important security principle is: never trust the frontend. Access, roles, company ownership, site ownership, API key scope, and file access must all be enforced server-side on every sensitive route. This phase also creates a credible baseline for future procurement, GDPR readiness, and later SOC 2-aligned controls.

Done looks like

All protected routes use a consistent auth middleware and reject unauthenticated access by default unless explicitly public. Session and auth cookie configuration is production-safe with httpOnly, secure, and appropriate sameSite settings. Password and login flows do not leak whether an email exists or not. Backend role checks exist and are enforced for admin-only and super-admin-only actions. Every tenant-owned read and write route enforces company_id ownership server-side. Every write that includes siteId validates that the site exists and belongs to the authenticated user’s company. Cross-company record access is blocked even when a valid ID is guessed or manually supplied. Agent and API keys are generated securely, shown only once, stored hashed, and support scopes, optional expiry, and revocation. Revoked or expired API keys fail auth immediately, while historical agent and audit logs remain intact. File uploads are validated by MIME type and extension, size-limited, stored privately, and retrievable only through authenticated authorised access. Security-relevant events are written to an audit trail, including login success and failure, role changes, subscription changes, API key creation and revocation, evidence upload and delete, and privileged admin actions. Login and other sensitive endpoints are rate-limited and return 429 under abuse conditions. Production responses do not expose stack traces or raw internal errors. Security headers are applied, including HSTS in production, X-Content-Type-Options, Referrer-Policy, and a baseline CSP. .env.example documents all required secrets and config values and no secrets are hardcoded in the repo. Backup and restore basics are documented, and health endpoints remain non-sensitive. Admin and super-admin users have a simple visibility surface for recent audit events and active and revoked API keys.

Scope

This phase includes authentication and session hardening, backend RBAC and privileged route protection, tenant and site ownership enforcement, agent and API key hardening, secure evidence and file upload and retrieval, audit logging for security-relevant events, rate limiting for sensitive endpoints, security headers and sanitised production error handling, and operational security documentation for secrets, backups, and recovery.

Non-goals

Do not block this phase on SSO or SAML, full MFA rollout, full SOC 2 policy pack, WAF or CDN automation, SIEM integration, advanced anomaly detection, customer-managed encryption keys, or full data residency controls. These can be layered on after the baseline is in place.

Implementation Requirements

Authentication and session hardening: Add or confirm a central requireAuth middleware used across all protected routes. Protected routes must default to requiring auth unless explicitly marked public. Session and auth cookies in production must use httpOnly true, secure true, and sameSite set to lax or stricter if compatible. Session expiry must be explicit and not effectively indefinite. Logout must invalidate the session or token state as applicable. Password storage must remain one-way hashed only. Login responses must be generic and not reveal whether the email exists. Password reset and invite tokens must be signed and time-limited. Login success and login failure must be audit logged. Suggested files include server/auth.ts, server/session-config.ts, and server/routes/auth-routes.ts. Acceptance criteria: unauthenticated request to a protected endpoint returns 401, login failure does not reveal whether email or password was wrong, production cookie config is secure, and login success and failure write audit entries.

Backend role-based access control: Add reusable guards requireRole(…roles) and requireSuperAdmin(). Apply these guards to company settings changes, team and role management, billing and subscription changes, framework configuration if admin-restricted, metric approval and rejection if role-restricted, and super-admin platform routes. Frontend hiding alone must not be treated as protection. Forbidden access must return 403. Suggested files include server/permissions.ts and server/middleware/require-role.ts. Acceptance criteria: non-admin cannot access admin-only endpoint, admin can access company-admin endpoint, and only super_admin can access platform-wide controls.

Tenant isolation and site ownership enforcement: Every tenant-owned query must be scoped by company_id. Never trust client-supplied companyId on ordinary routes. Any request containing siteId must validate that the site exists and belongs to the authenticated user company. Any record update or delete must also validate the target record belongs to the authenticated user company. Prefer DB queries scoped by both id and company_id, not insecure lookup-then-check where avoidable. Add shared helper functions so this logic is not reimplemented inconsistently. Suggested helpers include assertCompanyAccess(recordCompanyId, userCompanyId), validateSiteAccess(siteId, userCompanyId), and requireTenantRecord(resourceType, id, userCompanyId). Suggested file: server/tenant-access.ts, used across all route modules that handle company or site data. Acceptance criteria: user from Company A cannot fetch Company B record by ID, user from Company A cannot write data against Company B site, and site and company ownership checks work consistently across routes.

Agent and API key hardening: Generate API keys using cryptographically secure randomness. Show raw API key only once at creation time. Store only key_hash and key_prefix. Support fields name, scopes, created_by_user_id, created_at, last_used_at, expires_at nullable, and revoked_at nullable. Revoked or expired keys must fail auth immediately. Key list endpoints must never return raw key material. Historical agent runs and actions must remain after key revoke. Agent and webhook requests should support request signing verification if applicable. API key creation, usage on privileged routes, failure, and revocation must be audit logged. Suggested files include server/agent-auth.ts, server/agent-key-service.ts, and server/webhooks.ts. Acceptance criteria: raw key shown only on create, DB stores hash only, revoked key returns auth failure, missing scope returns 403, and historical logs remain after revoke.

Secure file upload and evidence protection: Validate uploads by both extension and MIME type. Reject executable or script-like file types. Enforce per-file max size and request size limits. Sanitise original filenames or replace with generated storage names. Store files privately, not in public static paths. File retrieval must require authenticated authorised access through signed URLs or a controlled backend streaming endpoint. File download and delete routes must enforce tenant ownership. Evidence files must inherit company and site ownership rules. If malware scanning is not yet implemented, add a forward-compatible scan_status field or document it as the immediate next phase. Suggested files include server/file-security.ts, server/evidence-routes.ts, and server/storage-service.ts. Acceptance criteria: disallowed file types rejected, oversized files rejected, user cannot access another company’s file by guessed path or key, and file retrieval requires auth plus ownership validation.

Audit logging and security event visibility: Capture at minimum login success, login failure, password reset and invite acceptance where applicable, role changes, team invite and remove actions, company settings changes, subscription changes, API key create and revoke, agent escalation and action events, metric approval and rejection, evidence upload and delete, and super-admin actions. Each event should include where possible actor_type user, agent, or system, actor_user_id or actor_agent_id, company_id, action, target_type, target_id, metadata, ip_address and user_agent when available, and created_at. Extend user_activities if it can support this cleanly, or add a dedicated audit_logs table. Suggested files include server/audit-log-service.ts and server/routes/admin-audit-routes.ts. Add a lean admin and super-admin audit view with recent events and filters by date, action, or user if practical. Keep it read-only and lightweight. Acceptance criteria: login failure produces audit entry, role change produces audit entry, admin and super-admin can view relevant audit history, and ordinary users cannot access platform-wide audit data.

Rate limiting and abuse protection: Rate-limit login endpoint strictly. Rate-limit password reset and invite resend endpoints strictly. Rate-limit uploads moderately. Rate-limit public or semi-public AI or agent endpoints where exposed. Return 429 on threshold breach. Ensure rate limiting does not unintentionally block trusted internal workers unless explicitly desired. Suggested file: server/rate-limit.ts, with middleware applied in relevant route modules. Acceptance criteria: repeated login attempts trigger 429, sensitive routes recover after configured window, and normal user flows remain functional.

Security headers and production-safe behaviour: Add security headers including Strict-Transport-Security in production, X-Content-Type-Options nosniff, Referrer-Policy, X-Frame-Options or CSP frame-ancestors, and a baseline Content-Security-Policy. Add centralised error handling that logs full internal detail server-side and returns sanitised client errors in production. Review /health and similar endpoints so they expose only safe operational status, not secrets or unnecessary internals. Suggested files include server/app-security.ts and server/error-handler.ts. Acceptance criteria: security headers present on production responses, production errors do not include stack traces, and health endpoint remains non-sensitive.

Secrets, env config, backup and restore documentation: No hardcoded secrets in repo. .env.example must document all required variables and their purpose. Separate secrets by concern where applicable, including session secret, JWT secret if used, webhook signing secret, email provider key, and storage credentials. Secret rotation must be possible without code changes. Add backup and restore documentation. Add deployment checklist covering HTTPS enabled, SSH key-only server access, firewalling and exposed ports, automated DB backups, and OS and package patching responsibility. Deliverables are updated .env.example, docs/security-foundation.md, and docs/backup-and-restore.md. Acceptance criteria: no live secrets in repo, docs clearly identify required env vars and recovery procedure, and backup and restore runbook is usable by an operator.

Data Model Changes

Add only where missing or incomplete. For audit_logs or equivalent extension to the existing activity table include id, company_id nullable, actor_type, actor_user_id nullable, actor_agent_id nullable, action, target_type nullable, target_id nullable, ip_address nullable, user_agent nullable, metadata jsonb, and created_at. For agent_api_keys confirm or add key_hash, key_prefix, scopes, expires_at, revoked_at, and last_used_at. For evidence or file metadata add original_filename, storage_key, mime_type, size_bytes, company_id, site_id nullable, and scan_status if implementing forward-compatible malware scan workflow.

Route and Middleware Summary

Add or harden middleware for requireAuth, requireRole(…roles), requireSuperAdmin(), assertCompanyAccess(…), validateSiteAccess(…), requireAgentAuth, requireAgentScope(scope), rateLimitLogin, rateLimitSensitiveEndpoints, and rateLimitUploads. Add or update services for auth-service, permission-service, tenant-access-service, audit-log-service, file-security-service, and agent-key-service.

UI and Product Surface

Keep UI changes lean and operational. For company admin, add a security section showing recent audit events, active API key count, and revoked API key history, plus API key management that allows key creation, scope selection, showing the raw key once, revoking the key, and showing last used timestamp. For super admin, add platform-wide audit visibility, recent failed auth attempts, recent privileged actions, and recent subscription and admin changes. Do not build a full security operations console in this phase.

Edge Cases and Explicit Rules

Super admin may have cross-company visibility; ordinary admins never do. Backend must ignore manipulated client-supplied companyId on ordinary routes. Revoked API keys must remain historically visible in logs. Deleting a user must not erase audit history; preserve actor references where possible. File deletion must not silently destroy auditability; define whether the file record is soft-deleted or marked unavailable. Health endpoints must expose only status, not sensitive environment detail. Rate limiting must not accidentally break internal schedulers or workers unless intentionally included.

Risks if skipped

Skipping this phase leaves the platform exposed to cross-tenant data leakage, unauthorised admin actions, leaked or abused API keys, public evidence or document exposure, brute force and login abuse, weak incident investigation capability, and avoidable procurement friction during security reviews.

Recommended implementation order

Implement in this order: auth and session hardening, backend RBAC, tenant and site isolation, agent and API key hardening, secure file access, audit logging, rate limiting, headers and sanitised errors, then docs and env and runbook updates. This order reduces material platform risk fastest.

Replit build instruction

Implement a platform-wide security foundation across authentication, permissions, tenant isolation, API key security, evidence file protection, audit logging, abuse protection, and production-safe HTTP behaviour. Use the existing Express and TypeScript architecture and current schema conventions. Prefer additive, low-risk refactors over broad rewrites. Reuse existing auth, storage, user activity, agent, and evidence or file structures where possible, but centralise security rules into shared middleware and helper services so enforcement is consistent. Deliver the work in clearly separated sections covering auth and session hardening, backend RBAC, company and site ownership enforcement, agent and API key security, secure file upload and retrieval, audit event logging and admin visibility, rate limiting, security headers and sanitised production errors, and docs and environment and runbook updates. For every protected resource, enforce ownership server-side. Do not rely on frontend hiding or client-sent tenant identifiers. Add focused tests for unauthenticated protected route denial, admin and super-admin route denial, cross-tenant read and write denial, revoked API key denial, missing-scope API key denial, and unauthorised file access denial.
