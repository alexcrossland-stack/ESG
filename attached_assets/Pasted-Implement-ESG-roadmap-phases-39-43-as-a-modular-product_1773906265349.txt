Implement ESG roadmap phases #39–#43 as a modular, production-safe extension of the existing platform.

This is an ESG management and framework-readiness system. It is NOT a formal compliance or certification engine. Use the language “readiness”, “alignment”, and “support”, not “compliance”, “certified”, or “fully compliant”, unless explicitly stated in existing code for unrelated legacy reasons.

Critical delivery rules

1. Preserve backwards compatibility
- Do not break existing metric entry flows, dashboards, reports, APIs, or current user data.
- Phase 1 must be introduced behind the existing abstraction where needed so current UX continues to work.
- Do not remove old structures until the new path is fully wired and verified.

2. Build reusable backend architecture
- Prefer structured schema, typed storage methods, calculation services, and reusable route handlers.
- Do not hardcode ESG logic into page components.
- Calculation and readiness logic must live in backend/service layers, not only the frontend.

3. Keep site-level support first-class
- All new ESG data models that represent business records must support `business_id` and `site_id` where relevant.
- `site_id` remains nullable.
- Rollups must be deliberate and configurable, not page-specific hacks.

4. Keep the product SME-friendly
- Core metrics remain default-on.
- Advanced capabilities should be progressively enabled without overwhelming the default experience.

5. Safe admin extensibility
- By Phase 5, admins should be able to evolve the ESG model without code changes.
- Avoid designs that require migrations for every new metric/framework adjustment.

PHASE #39 — ESG Phase 1: Metric Definitions & Calculation Engine
-----------------------------------

Objective

Refactor the current metric model into a proper `metric_definitions` architecture, seed 28 core metrics plus 30 advanced metrics, create structured `metric_values` and `metric_evidence`, and build a derived metric calculation engine with site-level rollup support.

Must deliver

- `metric_definitions` table
- `metric_values` table
- `metric_evidence` table
- `metric_calculation_runs` audit table (or clearly equivalent)
- seed data for 58 metrics total
- derived metric engine
- `/metrics-library` page
- evidence attachment support
- site-level rollup support
- no regression in existing metric flows

Schema requirements

Create `metric_definitions` with fields equivalent to:
- id
- code (stable unique identifier)
- name
- pillar
- category
- description
- data_type
- unit
- input_frequency
- is_core
- is_active
- is_derived
- formula_json
- framework_tags
- scoring_weight
- sort_order
- evidence_required
- rollup_method
- created_at
- updated_at

Create `metric_values` with fields equivalent to:
- id
- business_id
- site_id nullable
- metric_definition_id
- reporting_period_start
- reporting_period_end
- value_numeric nullable
- value_text nullable
- value_boolean nullable
- value_json nullable
- source_type (`manual`, `import`, `calculated`)
- status (`draft`, `submitted`, `approved`)
- notes
- entered_by_user_id
- created_at
- updated_at

Create `metric_evidence` with fields equivalent to:
- id
- metric_value_id
- file_url or storage_key
- file_name
- file_type
- uploaded_by_user_id
- uploaded_at
- notes

Create `metric_calculation_runs` with fields equivalent to:
- id
- business_id
- metric_definition_id
- site_id nullable
- reporting_period_start
- reporting_period_end
- status
- inputs_json
- output_json
- error_text nullable
- triggered_by_metric_value_id nullable
- created_at

Seed requirements

Seed:
- 28 existing core metrics
- 30 advanced metrics

Rules:
- core metrics: `is_core = true`, `is_active = true`
- advanced metrics: `is_core = false`, `is_active = false`
- derived metrics must use `is_derived = true`
- formulas must refer to stable metric codes, never display labels

Phase 1 UI

Build `/metrics-library`:
- grouped by pillar and category
- search/filter
- show core vs advanced
- show input frequency
- show evidence required
- advanced metric enable/disable toggle
- do not allow disabling core metrics in this phase unless already supported

Calculation engine requirements

Implement a dedicated calculation service in `server/calculations.ts` or equivalent:
- read `formula_json`
- resolve source dependencies by metric code
- calculate derived outputs for the same business / site / period context
- write derived values to `metric_values` using `source_type = calculated`
- log every run
- be idempotent
- prevent circular dependency loops
- handle missing source values safely
- trigger recalculation when a dependent source metric is created or updated

Rollup requirements

Support `rollup_method` values such as:
- `sum`
- `weighted_average`
- `latest`
- `none`

Rollups must work from site-level values to company-level aggregates where applicable.

Backward compatibility requirements

Mandatory:
- existing metric entry flows still work
- existing dashboards still load
- existing API consumers do not break
- introduce compatibility adapters in routes/storage if needed

Relevant files
- `shared/schema.ts`
- `server/calculations.ts`
- `server/storage.ts`
- `server/routes.ts`
- `client/src/pages`

PHASE #42 — ESG Phase 4: Multi-Dimensional Scoring & AI Assistant
-----------------------------------

Objective

Replace the single ESG score with a four-dimensional model and upgrade the assistant into an ESG advisor layer.

Must deliver

Dashboard score outputs:
- Core ESG Completeness
- ESG Performance
- Management Maturity
- Framework Readiness

Assistant capabilities:
- explain metrics
- identify missing submissions and evidence
- identify missing policies / targets / owners
- recommend next actions
- summarise readiness in plain language
- suggest draft ESG action plans

Scoring requirements

Implement four separate backend scoring services:

1. Completeness
- expected data vs submitted data for current period

2. Performance
- metric values vs targets and/or prior period trends

3. Management Maturity
- factors in policies, ownership, targets, actions, evidence, review cycles

4. Framework Readiness
- covered / partial / missing requirements by selected framework

Important:
- scoring logic belongs in backend/service layer
- frontend only renders outputs and explanations

Dashboard UI

Update dashboard to show:
- four score cards or gauges
- plain-language explanations
- per-framework readiness if selected

Assistant upgrade

Extend assistant logic so it can:
- explain metric definitions
- recognise framework selections
- interpret readiness gaps
- detect missing evidence
- recommend policies, targets, actions
- highlight overdue reviews and unassigned ownership
- use “alignment/readiness” language only

Relevant files
- `server/calculations.ts`
- `server/routes.ts`

IMPLEMENTATION EXPECTATIONS
-----------------------------------

Use phased delivery. Complete one phase cleanly before starting the next.

For each phase, provide:
1. schema changes
2. storage layer changes
3. route additions/changes
4. frontend pages/components
5. seed data additions
6. migration/backwards-compatibility notes
7. any assumptions made

Important architecture expectations

- Use stable codes for metrics and framework requirements
- Avoid hardcoded ESG logic in React pages
- Prefer typed helpers and service-layer functions
- Keep nullable `site_id`
- Enforce business/site ownership validation on writes
- Keep derived metrics non-editable by end users
- Design formulas and mappings so future admin UI can manage them safely

Deliver Phase 1 first.
Do not start Phase 2 until Phase 1 is complete and stable.
