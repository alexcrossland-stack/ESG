What & Why

Enable a non-technical SME user to complete a full activation journey in one session (10–15 minutes):

Sign up → complete onboarding → add first data → understand outcome → generate first report

This is not a feature build. It is a system-wide activation, usability, copy, and flow optimisation pass.

Core rule:
Every screen must answer:
	1.	What is this?
	2.	Why does it matter?
	3.	What should I do next?

If a user can land on a page and do nothing, the task is incomplete.

This task supersedes #5, #6, #7, #19, #24.

⸻

Critical Architecture Rule (must follow)

Use ONE shared activation state model.

Do NOT duplicate activation logic across pages.

Create a single derived state:

activationState = {
hasCompletedOnboarding: boolean,
hasAddedData: boolean,
hasUploadedEvidence: boolean,
hasGeneratedReport: boolean
}

This must be the single source of truth for:
	•	dashboard checklist
	•	next-step banner
	•	empty state CTAs
	•	report readiness

Implement via shared hook or context (e.g. useActivationState).

⸻

Done Looks Like
	•	User completes full journey in one session with no confusion
	•	Dashboard shows persistent checklist (never disappears until complete)
	•	Checklist shows live progress (0–100%)
	•	Next-step banner always visible until activation complete
	•	No empty charts, blank tables, or unexplained “0” values anywhere

Onboarding:
	•	Plain English steps with “what” + “why”
	•	Defaults applied automatically
	•	First site auto-created
	•	Completion screen → success + summary + CTA “Add your first data point”

Data Entry:
	•	Units + example values
	•	Inline validation
	•	Success feedback + next-step CTA
	•	“Metric category” → “Type of data” + explanation

Empty States (ALL pages):
	•	what it is
	•	why it matters
	•	what to do next
	•	one CTA (links directly to action)

Reports:
	•	Empty state + “Generate report” CTA
	•	Framework explained in plain English
	•	Insufficient data message explains EXACTLY what is missing + link to fix

Tooltips:
	•	framework, evidence, maturity, scope 1/2/3, type of data
	•	one sentence, plain English, click-based

Error Handling:
	•	no generic errors
	•	every error = cause + fix
	•	no silent failures

Micro UX:
	•	“Submit” → Save data
	•	“Continue” → Next step
	•	loading states on all async actions
	•	no dead links

Analytics:
Track:
	•	onboarding_step_completed
	•	onboarding_completed
	•	first_data_added
	•	first_report_generated
	•	onboarding_drop_off

Endpoint:
POST /api/analytics/events

Payload:
	•	timestamp
	•	user_id
	•	company_id
	•	event_name

	•	new data models/tables (unless unavoidable)

	•	admin-side changes
