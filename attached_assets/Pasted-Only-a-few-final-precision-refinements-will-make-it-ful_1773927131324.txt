#44 - Identity Hardening, SSO Readiness & GDPR

What & Why

Extend the security foundation into identity maturity and regulatory readiness. Introduces TOTP-based MFA with enforcement policies, prepares the auth architecture for future SSO/SAML integration without overbuilding, and delivers GDPR-aligned data rights tooling (export, deletion, consent traceability).

This reduces account takeover risk, makes enterprise onboarding feasible, and ensures legal defensibility. Builds directly on the auth, RBAC, tenant isolation, and audit logging established in Task #12.

⸻

Done looks like

Users can enable TOTP MFA via an authenticator app; backup/recovery codes are generated (shown once, hashed at rest)

MFA secrets are stored encrypted at rest using a server-side encryption key (env-managed); backup codes are stored as one-way hashes

MFA is enforced at login once enabled; recovery flow allows backup code as fallback (single-use, invalidated after use and audit logged)

Company admins can set an MFA policy: optional / admin-required / all-required

When MFA is required by policy, users must complete MFA setup immediately after password authentication before accessing the platform

Identity provider model exists in the database (SAML/OIDC-ready); user model supports external_id and identity_provider_id while retaining email as the primary unique identifier

Auth service separates identity verification from session creation so additional providers can be added without refactoring login flow

Stub SSO endpoints exist (initiate + callback) with no regression to existing password login

Company admin can trigger a structured data export (JSON bundle) covering metrics, reports, evidence metadata, and user/account data — excluding secrets

Users can request their own personal data export via the settings UI

Export jobs are logged, processed asynchronously, and provide a secure expiring download link (time-limited, access-controlled to the requesting user/company, single-use or short TTL)

Export files are automatically deleted after expiry

Users can request account deletion; personal data is anonymised/removed while audit logs retain an anonymised actor reference

Company admins can initiate a staged company deletion (mark pending → delay window → cascade delete/anonymise), requiring explicit confirmation (e.g. typing company name)

Deletion operations are idempotent and safe to retry

User acceptance of terms and privacy policy is recorded with timestamp and version on signup and after policy updates

Admin UI shows: MFA policy control, per-user MFA status, export trigger, company deletion controls (with confirmation safeguard)

User settings show: MFA enable/disable, backup code view/regeneration, personal export request, account deletion request

All MFA actions (including backup code usage as a security-sensitive event), export requests, deletion requests, and SSO config changes are captured in the audit log
