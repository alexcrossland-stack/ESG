I want to keep the current ESG Manager product structure, but I need the app upgraded from a good MVP into a more robust production-ready SME SaaS platform. Please implement the following improvements in priority order.

1. Replace the current authentication and security model

The current auth approach is not strong enough. Update it as follows:
	•	Replace SHA-256 password hashing with Argon2id or bcrypt
	•	Use a proper per-user salt handled by the hashing library
	•	Stop using a fixed salt
	•	Replace the current token approach with a more secure session/auth design
	•	Prefer secure HTTP-only cookies for auth if practical
	•	If token auth remains, add:
	•	token expiry
	•	token revocation
	•	secure logout
	•	server-side session invalidation
	•	Add rate limiting on login, register, and password reset endpoints
	•	Add password reset flow
	•	Add email verification flow
	•	Add basic session management so users can be logged out securely
	•	Add audit logging for:
	•	login success
	•	login failure
	•	password reset
	•	email verification
	•	logout
	•	permission changes

I want all auth and security code refactored so it is appropriate for a real SaaS application, not just a demo environment.

⸻

2. Introduce stronger user roles and permissions

The current two-role model of admin and editor is too limited. Please replace it with a more flexible role and permissions system.

Implement these minimum roles:
	•	Admin — full company access, settings, users, templates, reports, approvals
	•	Contributor — can input data, edit assigned metrics, update questionnaires, draft policies
	•	Approver — can review and approve policies, reports, questionnaire answers, and locked submissions
	•	Viewer — read-only access to dashboards, reports, and approved documents

Also add permission controls by module where practical, for example:
	•	metrics data entry
	•	policy editing
	•	report generation
	•	questionnaire access
	•	settings admin
	•	template admin
	•	user management

I want permissions enforced both in the UI and on the backend API.

⸻

3. Add approval states and governance workflows

I want the app to distinguish clearly between draft content and approved content. Please add workflow states across the platform.

Implement standard statuses such as:
	•	draft
	•	submitted
	•	locked
	•	approved
	•	rejected
	•	archived

Apply these states to at least:
	•	metric values
	•	raw data submissions
	•	generated policies
	•	questionnaire answers
	•	reports

Examples of behaviour I want:
	•	contributors can save drafts
	•	approvers can approve or reject submissions
	•	approved items become read-only unless reopened by an admin
	•	locked periods cannot be edited except by admin override
	•	each approval/rejection should store:
	•	who did it
	•	date/time
	•	comments/reason

This should create a proper governance trail for ESG data and outputs.

4. Make AI outputs auditable and controlled

The AI tools are useful, but I need them to be safer and more defensible.

Please update all AI features so that:
	•	every AI-generated output is marked as Draft – Review Required
	•	every generation stores:
	•	model name/version
	•	prompt version
	•	generation timestamp
	•	user who triggered it
	•	source data used
	•	final output
	•	users can view the questionnaire inputs or internal company data that informed the output
	•	AI outputs cannot be treated as approved until a human reviews and approves them
	•	questionnaire autofill must show:
	•	suggested answer
	•	rationale
	•	supporting company data used
	•	confidence level or completeness indicator
	•	if there is insufficient data, the AI should say so instead of inventing an answer

Also add prompt protection and validation for pasted supplier questionnaires so the system is less vulnerable to bad instructions inside pasted text.

I want the AI layer designed as an assisted drafting tool, not an uncontrolled auto-answer engine.

⸻

Evidence should be a core part of the platform rather than just file upload storage.

Please implement a proper evidence workflow with these features:
	•	allow evidence files to be attached to:
	•	metric values
	•	raw data submissions
	•	policies
	•	questionnaire answers
	•	reports
	•	store metadata for each evidence file:
	•	uploaded by
	•	upload date
	•	linked module
	•	linked period
	•	file type
	•	description
	•	review status
	•	expiry/review date where relevant
	•	add evidence statuses such as:
	•	uploaded
	•	reviewed
	•	approved
	•	expired
	•	show evidence coverage in the UI, for example:
	•	which metrics have evidence
	•	which periods are missing evidence
	•	which evidence is out of date

I also want report exports and questionnaire outputs to be able to show whether the answer or metric is evidenced, estimated, or manually declared.

6. Improve the scoring and dashboard logic

The traffic light system is useful, but the overall ESG score needs more control.

Please update the scoring engine so that:
	•	scoring supports weighting by:
	•	category
	•	metric importance
	•	material topics
	•	inactive modules do not unfairly penalise users
	•	missing data is handled explicitly and does not silently distort the score
	•	compliance yes/no metrics are weighted separately from continuous metrics where needed
	•	target-range metrics are scored correctly
	•	calculated and derived metrics can be flagged separately from manual values

On the dashboard, add:
	•	missing data alerts
	•	overdue action alerts
	•	upcoming policy review dates
	•	evidence coverage indicator
	•	percentage of metrics submitted for current period
	•	clear explanation of how the ESG score is calculated

I want the dashboard to feel more credible and less like a generic traffic-light summary.

7. Strengthen the carbon calculator

The current carbon calculator is a good start, but I want it upgraded into a more reliable SME carbon module.

Please implement the following:
	•	store emission factor sets with versioning
	•	make factor year visible in the UI and exports
	•	add methodology notes to every calculation
	•	distinguish:
	•	actual values
	•	estimated values
	•	proxy values
	•	add optional fuel-type selection where relevant
	•	support both simple activity-based and fallback proxy-based calculations where data is incomplete
	•	ensure calculations are rules-based in code, not AI-generated
	•	show assumptions clearly in outputs and exports

I do not want the app positioned as full enterprise carbon accounting, but it should be a trustworthy SME carbon estimator.
