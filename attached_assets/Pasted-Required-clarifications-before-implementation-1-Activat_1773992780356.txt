Required clarifications before implementation

1. Activation state must be derived, not stored separately

Do not add a new persistent activation table unless absolutely unavoidable.

activationState should be derived from existing platform state wherever possible, for example:
	•	onboarding completion flag
	•	presence of at least one saved data entry
	•	presence of at least one evidence upload
	•	presence of at least one generated report

This is to avoid state drift between “real user progress” and “activation progress”.

2. Checklist and banner must never contradict each other

The dashboard checklist, next-step banner, and report readiness messaging must all use the same shared activation source.

It must not be possible for:
	•	checklist to show “data complete”
	•	while banner still says “add your first data point”

Any such mismatch is a fail.

3. Empty-state CTAs must link to the exact next action

Do not route users to generic landing pages if a direct action route exists.

Examples:
	•	“Add your first data point” should open the correct data-entry flow
	•	“Upload evidence” should open the evidence upload flow
	•	“Generate report” should take the user directly into report generation

Generic navigation that still leaves the user deciding what to do is not sufficient.

4. First-time analytics events must only fire once per company/user journey

For:
	•	first_data_added
	•	first_report_generated

Ensure these do not fire repeatedly on every later save or report generation. They must represent the first successful completion of that milestone.
