Please implement the Beta / Tester Access System exactly as follows.

GOAL
Allow selected companies to access all Pro features during pre-launch beta without creating fake Stripe subscriptions. Beta access must be a separate override layer and remain fully independent of Stripe billing.

ARCHITECTURAL DECISION
This must be implemented at the company level, not the user level, because:
- planTier is stored on companies
- Stripe webhooks update companies
- feature gating reads company.planTier
- user-level beta flags would create inconsistent access logic

Use a user's email only as the lookup method to find their companyId, then apply beta access to that company.

1. DATABASE CHANGES
In shared/schema.ts, add these nullable / defaulted fields to the companies table:

- isBetaCompany: boolean("is_beta_company").default(false)
- betaExpiresAt: timestamp("beta_expires_at")
- betaAccessLevel: text("beta_access_level")
- betaGrantedBy: text("beta_granted_by")
- betaReason: text("beta_reason")

Then run:
npm run db:push

No destructive changes. Only additive schema changes.

2. ACCESS LOGIC
In server/routes.ts, add this helper:

function getEffectivePlanTier(company: Company): { tier: "free" | "pro"; isBeta: boolean } {
  if (company.planTier === "pro") return { tier: "pro", isBeta: false };

  const now = new Date();
  if (
    company.isBetaCompany &&
    (!company.betaExpiresAt || company.betaExpiresAt > now)
  ) {
    return { tier: "pro", isBeta: true };
  }

  return { tier: "free", isBeta: false };
}

Use this helper anywhere the effective plan tier is needed.

Update GET /api/billing/status so it:
- uses getEffectivePlanTier(company)
- returns effective planTier
- returns isBeta
- returns betaExpiresAt

Important:
- Do not modify Stripe checkout logic
- Do not modify Stripe webhook behavior
- Do not create fake Stripe subscriptions
- Stripe remains the source of truth for paid subscriptions only
- Beta access is only an override layer for feature access

3. ADMIN ENDPOINTS
In server/routes.ts, add two secure endpoints protected by:
- requireAuth
- requireSuperAdmin

A) POST /api/admin/beta/grant

Request body:
{
  "email": "user@example.com",
  "expiresInDays": 30,
  "accessLevel": "pro",
  "reason": "pre-launch testing"
}

Behavior:
- find the user by email
- verify the user exists
- verify the user has a companyId
- load the company
- set:
  - isBetaCompany = true
  - betaExpiresAt = now + expiresInDays
  - betaAccessLevel = accessLevel ?? "pro"
  - betaGrantedBy = current super admin email
  - betaReason = reason ?? null
- write an audit log entry to super_admin_actions with:
  - action = "beta_access_granted"
  - details = { email, expiresInDays, reason, companyId }
- return:
  {
    success: true,
    companyId,
    betaExpiresAt
  }

B) POST /api/admin/beta/revoke

Request body:
{
  "email": "user@example.com"
}

Behavior:
- find the user by email
- verify user and company exist
- clear:
  - isBetaCompany = false
  - betaExpiresAt = null
  - betaAccessLevel = null
  - betaGrantedBy = null
  - betaReason = null
- write audit log entry:
  - action = "beta_access_revoked"
  - details = { email, companyId }
- return:
  {
    success: true
  }

Error handling:
- return clear 400 errors for invalid input
- return clear 404 errors when user or company is not found
- do not fail silently
